# Image Upload Feature - API Parameters Documentation

This document defines all API request/response parameters, type definitions, error codes, and validation rules for the image upload feature. It follows Domain-Driven Design (DDD) principles with clear separation between domain types and API contract types.

---

## 1. UPLOAD IMAGE ENDPOINT

### Endpoint Details

**URL:** `POST /api/images/upload`

**Authentication:** Required (JWT Bearer Token)

**Content-Type:** `multipart/form-data`

**Authorization Header:** `Authorization: Bearer {jwt_token}`

---

### 1.1 Request Parameters

#### Request Type: `IUploadImageRequest`

```typescript
export interface IUploadImageRequest {
  image: File;
  userId: string;
}
```

#### Parameter Definitions

| Parameter | Type | Required | Description | Validation |
|-----------|------|----------|-------------|-----------|
| `image` | File | Yes | Image file to upload | Must be valid image file (JPEG, PNG, WebP) |
| `userId` | string | Yes | User ID from JWT token | Non-empty string, extracted from token |

#### Validation Rules

**image parameter:**
- Must be a File object (not null or undefined)
- MIME type must be one of: `image/jpeg`, `image/png`, `image/webp`
- File size must be between 1KB and 10MB (1,024 - 10,485,760 bytes)
- File must not be empty
- File extension must match MIME type (magic number validation)

**userId parameter:**
- Must be a non-empty string
- Must match the userId in the JWT token (backend validation)
- Used to organize images in GCP: `{userId}/{imageId}/{filename}`

---

### 1.2 Response Parameters (Success)

#### Response Type: `IUploadImageResponse`

```typescript
export interface IUploadImageResponse {
  success: true;
  imageId: string;
  gcpUrl: string;
}
```

#### Parameter Definitions

| Parameter | Type | Description | Example |
|-----------|------|-------------|---------|
| `success` | boolean | Always `true` for success | `true` |
| `imageId` | string | Unique identifier for uploaded image | `"img_1a2b3c4d5e6f"` |
| `gcpUrl` | string | Public/signed URL to access image in GCP | `"https://storage.googleapis.com/bucket/userId/imageId/filename"` |

#### Response Details

- **HTTP Status Code:** 200 OK
- **Content-Type:** `application/json`
- **imageId:** Unique identifier generated by backend, used for retrieval and metadata operations
- **gcpUrl:** Signed URL valid for 1 hour (configurable), allows direct access to image in GCP Cloud Storage

---

### 1.3 Response Parameters (Error)

#### Response Type: `IUploadImageErrorResponse`

```typescript
export interface IUploadImageErrorResponse {
  success: false;
  error: {
    message: string;
    code: string;
  };
}
```

#### Parameter Definitions

| Parameter | Type | Description |
|-----------|------|-------------|
| `success` | boolean | Always `false` for error |
| `error.message` | string | User-friendly error message |
| `error.code` | string | Machine-readable error code |

---

### 1.4 Error Codes and HTTP Status Codes

| Error Code | HTTP Status | Message | Cause | User Action |
|-----------|-------------|---------|-------|-------------|
| `INVALID_IMAGE_FORMAT` | 400 | "Invalid image format. Please upload JPG, PNG, or WebP." | File MIME type not supported | Choose supported image format |
| `IMAGE_SIZE_EXCEEDS_LIMIT` | 400 | "File size exceeds 10MB limit. Please choose a smaller file." | File size > 10MB | Compress or choose smaller image |
| `IMAGE_SIZE_BELOW_MINIMUM` | 400 | "File is too small. Minimum size is 1KB." | File size < 1KB | Choose valid image file |
| `EMPTY_FILE` | 400 | "File is empty. Please choose a valid image." | File has 0 bytes | Choose valid image file |
| `UNAUTHORIZED` | 401 | "Authentication required. Please log in." | Missing or invalid JWT token | Log in with phone number and OTP |
| `GCP_UPLOAD_FAILED` | 503 | "Upload failed. Please try again later." | GCP Cloud Storage error | Retry upload |
| `METADATA_STORAGE_FAILED` | 500 | "Failed to save image metadata. Please try again." | MongoDB storage error | Retry upload |

## 2. RETRIEVE IMAGE ENDPOINT

### Endpoint Details

**URL:** `GET /api/images/{imageId}`

**Authentication:** Required (JWT Bearer Token)

**Content-Type:** Varies (image/jpeg, image/png, image/webp)

**Authorization Header:** `Authorization: Bearer {jwt_token}`

---

### 2.1 Request Parameters

#### URL Path Parameters

| Parameter | Type | Required | Description | Validation |
|-----------|------|----------|-------------|-----------|
| `imageId` | string | Yes | Unique image identifier | Non-empty string, must exist in database |

#### Query Parameters

None

---

### 2.2 Response Parameters (Success)

#### Response Type: Image File Stream

```typescript
export interface IRetrieveImageResponse {
  // Binary image data
  // Headers:
  // - Content-Type: image/jpeg | image/png | image/webp
  // - Content-Length: number
  // - Cache-Control: public, max-age=3600
  // - ETag: string (for caching)
}
```

#### Response Details

- **HTTP Status Code:** 200 OK
- **Content-Type:** Determined by original image MIME type
- **Content-Length:** Size of image file in bytes
- **Cache-Control:** `public, max-age=3600` (1 hour browser cache)
- **ETag:** Hash of image for cache validation
- **Body:** Binary image file data

---

### 2.3 Response Parameters (Error)

#### Error Responses

| HTTP Status | Error Code | Message | Cause |
|-------------|-----------|---------|-------|
| 404 | `IMAGE_NOT_FOUND` | "Image not found." | Image ID doesn't exist or was deleted |
| 401 | `UNAUTHORIZED` | "Authentication required. Please log in." | Missing or invalid JWT token |
| 403 | `FORBIDDEN` | "You don't have permission to access this image." | User doesn't own the image |
| 503 | `GCP_RETRIEVAL_FAILED` | "Failed to retrieve image. Please try again." | GCP Cloud Storage error |

## 5. VALIDATION RULES

### 5.1 Image Format Validation

**Supported Formats:**
- JPEG: `image/jpeg` (extensions: `.jpg`, `.jpeg`)
- PNG: `image/png` (extension: `.png`)
- WebP: `image/webp` (extension: `.webp`)

**Validation Strategy:**
1. **MIME Type Check:** Verify Content-Type header matches supported formats
2. **Extension Check:** Verify file extension matches supported formats
3. **Magic Number Check:** Verify file content matches extension (prevent spoofing)

**Magic Numbers (File Signatures):**
- JPEG: `FF D8 FF` (first 3 bytes)
- PNG: `89 50 4E 47` (first 4 bytes)
- WebP: `52 49 46 46` + `57 45 42 50` (bytes 0-3 and 8-11)

**Validation Function (Domain Layer):**
```typescript
export function validateImageFormat(mimeType: string, fileBuffer: Buffer): boolean {
  const supportedMimeTypes = ['image/jpeg', 'image/png', 'image/webp'];
  
  if (!supportedMimeTypes.includes(mimeType)) {
    return false;
  }
  
  // Verify magic number matches MIME type
  return verifyMagicNumber(mimeType, fileBuffer);
}
```

---

### 5.2 Image Size Validation

**Size Constraints:**
- **Minimum Size:** 1KB (1,024 bytes)
- **Maximum Size:** 10MB (10,485,760 bytes)

**Validation Locations:**
- **Frontend:** Pre-upload validation for immediate user feedback
- **Backend:** Final validation before storage (security)

**Validation Function (Domain Layer):**
```typescript
export function validateImageSize(fileSize: number): boolean {
  const MIN_SIZE = 1024; // 1KB
  const MAX_SIZE = 10485760; // 10MB
  return fileSize >= MIN_SIZE && fileSize <= MAX_SIZE;
}

export function validateImageNotEmpty(fileSize: number): boolean {
  return fileSize > 0;
}
```

---

### 5.3 Filename Validation and Sanitization

**Sanitization Rules:**
- Remove path traversal attempts (`../`, `..\\`)
- Remove special characters except: `-`, `_`, `.`
- Preserve original file extension
- Limit filename length to 255 characters
- Convert to lowercase for consistency

**Sanitization Function (Domain Layer):**
```typescript
export function sanitizeFilename(filename: string): string {
  // Remove path traversal
  let sanitized = filename.replace(/\.\.\//g, '').replace(/\.\.\\/g, '');
  
  // Remove special characters except - _ .
  sanitized = sanitized.replace(/[^a-zA-Z0-9._-]/g, '_');
  
  // Limit length
  if (sanitized.length > 255) {
    const ext = sanitized.substring(sanitized.lastIndexOf('.'));
    sanitized = sanitized.substring(0, 255 - ext.length) + ext;
  }
  
  // Convert to lowercase
  sanitized = sanitized.toLowerCase();
  
  return sanitized;
}
```

---

### 5.4 User ID Validation

**Validation Rules:**
- Must be non-empty string
- Must match userId in JWT token (backend validation)
- Used for access control and data organization

**Validation Function (Domain Layer):**
```typescript
export function validateUserId(userId: string): boolean {
  return userId && userId.trim().length > 0;
}

export function validateUserOwnership(userId: string, tokenUserId: string): boolean {
  return userId === tokenUserId;
}
```

---

### 6.3 Error Handling Strategy

**Frontend Error Handling:**
1. Check `success` field in response
2. If `success === false`, extract error code and message
3. Display user-friendly message to user
4. Log error code for debugging
5. Provide retry option for service errors (5xx)

**Backend Error Handling:**
1. Validate all inputs before processing
2. Throw specific error types (e.g., `InvalidImageFormatError`)
3. Catch errors in application layer
4. Map errors to appropriate HTTP status codes
5. Return standardized error response
6. Log error with context for debugging

## 7. AUTHENTICATION & AUTHORIZATION

### 7.1 JWT Token Requirements

**Token Format:** `Bearer {jwt_token}`

**Token Location:** `Authorization` header

**Token Contents:**
```typescript
{
  userId: string;
  phoneNumber: string;
  iat: number; // Issued at
  exp: number; // Expiration time
}
```

**Token Validation:**
- Must be present in Authorization header
- Must be valid JWT signature
- Must not be expired
- Must contain userId claim

**Example Header:**
```
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOiJ1c2VyXzEyMzQ1Njc4OTAiLCJwaG9uZU51bWJlciI6IisxMjM0NTY3ODkwIiwiaWF0IjoxNjA0NzI5NzAwLCJleHAiOjE2MDQ3MzMzMDB9.signature
```

---

### 7.2 Authorization Rules

**Image Upload:**
- User must be authenticated (valid JWT token)
- userId in request must match userId in token
- User can only upload images for themselves

**Image Retrieval:**
- User must be authenticated (valid JWT token)
- User can only retrieve images they own
- Backend verifies userId matches token userId

**Image Metadata:**
- User must be authenticated (valid JWT token)
- User can only access metadata for images they own
- Backend verifies userId matches token userId
